## Project name
Morsegrid Outfitters — AI Revenue Recovery Agent

## Elevator pitch (one line)
A multi-agent system on Gemini 3 + Google ADK that recovers lost ecommerce revenue from
both abandoned carts and dormant customers, reasoning over live data through the MongoDB
MCP server.

---

## Inspiration
Stores leak revenue from two leaks at once: ~70% of carts are abandoned, and a long tail of
past customers quietly goes dormant. The standard fix — a fixed Klaviyo/Postscript drip that
fires the same template at everyone — ignores *why* each person left. We wanted an agent that
actually reasons over each shopper's history and the live catalog and writes something a real
store employee would write.

## What it does
Three cooperating Gemini 3 agents work one ranked queue that mixes two opportunity types:
- **Abandoned carts** — reassure about the exact item, suggest in-stock alternatives if it
  sold out, or "complete the look" for multi-item carts.
- **Dormant customers** — warm win-backs tied to a real reason (a new arrival matching their
  history, or the item they once searched for is finally in stock).

A deterministic **Scorer** ranks every opportunity by expected recovered margin, then **three
Gemini 3 agents** take over: a **Strategist** reasons over that ranked queue and checks contact
fatigue (via MongoDB MCP) to set today's priorities; a **Nurturer** (MCP history + Atlas Vector
Search) drafts a personalized message; a human approves it in a Streamlit dashboard; and a
**Sender** picks the best channel (email/SMS/IG), delivers, and logs the result via MCP. It even
switches an email-dead shopper to SMS automatically. *(A sidebar toggle can swap the Strategist
and Sender for fast deterministic versions.)*

## How we built it
- **Gemini 3 Flash** on **Vertex AI** for all agent reasoning and drafting.
- **Google ADK** (Agent Builder, code-first) orchestrating three agents — **Strategist,
  Nurturer, Sender** — and their tools.
- **Hybrid by design:** a deterministic Scorer does the EV arithmetic (auditable, reproducible);
  the agents apply *judgment*. We use the LLM where reasoning adds value, code where it's a formula.
- **MongoDB MCP Server** as the agents' *only* runtime database interface (`find`,
  `aggregate`, `count`, `insert-many`).
- **MongoDB Atlas Vector Search** over Vertex `text-embedding-004` for product matching.
- A **Shopify-style webhook** (`webhook_server.py`) normalizes `checkouts/abandoned` payloads
  into a storefront-agnostic schema, so any ecommerce platform can plug in.
- **Streamlit** for the human-in-the-loop dashboard; **Resend** for real email; deployed on
  **Google Cloud Run**.

## Technologies used
Gemini 3 Flash, Google ADK, Vertex AI, Google Cloud Run, MongoDB Atlas, MongoDB MCP Server,
Atlas Vector Search, text-embedding-004, Streamlit, Resend, Shopify webhook format.

## Data sources
100% synthetic demo data for a fictional motorcycle-gear store: 50 products, 18 customers,
~180 behavior events, 6 abandoned carts. Generated by `data/seed_data.py`; embedded and
indexed by `data/build_index.py`. Real Shopify/WooCommerce stores plug in via the webhook.

## Challenges
- Doing vector search *through* an agent without making the LLM hand-type 768-float vectors —
  solved with an `embed_query` + `$vectorSearch` tool that the agent calls by intent.
- Keeping two very different opportunity types (carts decay in hours; dormancy in weeks)
  comparable in a single ranked queue — solved by scoring both in expected-margin dollars.
- Running the Node-based MongoDB MCP server alongside Python in one Cloud Run container.

## What we learned
Routing all agent DB access through MCP made the agent dramatically simpler to reason about
and audit — every tool call is visible in the trace. The expected-value Planner is what turns
"send everyone an email" into a defensible business decision.

## What's next
Real Twilio SMS + Meta IG, a holdout group for measured incremental-revenue lift, and an
autonomous mode that sends low-risk recoveries without human review.
